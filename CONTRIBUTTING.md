# チーム開発の進め方とルール

このプロジェクトに貢献していただきありがとうございます！
円滑なチーム開発のために、以下のルールとガイドラインを守るようご協力をお願いします。

## 💬 基本的なコミュニケーション

- **タスク管理:** すべてのタスクは [GitHub Issues](<あなたのGitHub IssuesのURL>) で管理します。作業を始める前に、必ずIssueが作られているか確認しましょう。
- **相談:** 実装で迷ったり、技術的な問題に直面した場合は、一人で抱え込まずに気軽に相談しましょう。

## 🌿 ブランチ戦略 (GitHub Flow)

このプロジェクトでは、シンプルで分かりやすい「GitHub Flow」を採用します。

1.  **`main`ブランチは常に正常**
    `main`ブランチは、常にデプロイ可能（＝正常に動作する）な状態を保ちます。`main`ブランチに直接コミットすることは禁止です。

2.  **作業は必ず新しいブランチで**
    新しい機能の開発やバグ修正を行う際は、必ず`main`ブランチから最新の状態を取得し、新しいブランチを作成します。

    ```bash
    # mainブランチを最新の状態にする
    git checkout main
    git pull origin main

    # 新しいブランチを作成して移動する
    git checkout -b <ブランチ名>
    ```

3.  **分かりやすいブランチ名**
    ブランチ名は、そのブランチの目的が簡潔に分かるように命名します。
    - **プレフィックス:** `feature/`, `fix/`, `docs/` などを付けます。
    - **命名規則:** `プレフィックス/目的`
    - **例:**
        - `feature/user-login-page` （ユーザーログインページの開発）
        - `fix/header-layout-bug` （ヘッダーのレイアウト崩れの修正）
        - `docs/update-readme` （READMEの更新）

## ✅ コミットメッセージの書き方

後から履歴を追いやすくするために、コミットメッセージの先頭に、そのコミットの種類を表す`type`を付けましょう。

- **`feat:`**: 新機能の追加
- **`fix:`**: バグの修正
- **`docs:`**: ドキュメントの変更（READMEなど）
- **`style:`**: コードのフォーマット変更（Prettierによる整形など）
- **`refactor:`**: リファクタリング（機能的な変更はないコードの改善）
- **`test:`**: テストコードの追加・修正

**コミットメッセージの例:**

```
feat: ユーザー新規登録APIのエンドポイントを追加
fix: 商品一覧画面で価格が正しく表示されない問題を修正
docs: セットアップ手順をREADMEに追記
```

## Pull Request (PR) の作成

作業が完了し、`main`ブランチにマージしてほしくなったら、GitHub上でPull Requestを作成します。

1.  **セルフレビューを行う**
    PRを作成する前に、自分の変更内容に不要なファイル（`console.log`など）が含まれていないか、コーディング規約に従っているかなどを自分で一度確認します。

2.  **分かりやすいPRのタイトルと説明**
    - **タイトル:** 何を変更したのかが簡潔に分かるように書きます。（例: `ユーザー登録機能の追加`）
    - **説明:**
        - **目的:** なぜこの変更が必要だったのか。（例: `Closes #12` のように関連するIssue番号を書くと自動でリンクされます）
        - **変更内容:** 具体的に何をしたのかを箇条書きなどで記述します。
        - **確認方法:** レビュアーがどうすれば動作確認できるかを書きます。（例: `1. /register にアクセス` `2. フォームを入力して登録ボタンを押す`）

3.  **WIP (Work In Progress) の活用**
    まだ作業途中でレビューは不要だけど、他のメンバーに進捗を共有したい場合は、**Draft Pull Request**として作成しましょう。

## 🤝 コードレビュー

PRが作成されたら、もう一人のメンバーがコードレビューを行います。

- **レビュアーの役割:**
    - 実装が要件を満たしているか確認する。
    - より良い書き方や改善点があれば提案する。
    - バグや潜在的な問題がないか探す。

- **レビューの心構え:**
    - 敬意を持って、建設的なフィードバックを心がけましょう。
    - 指摘だけでなく、良いと思ったコードは積極的に褒めましょう！
    - 修正依頼があった場合、PRの作成者は感謝を持って対応し、修正後に再度レビューを依頼します。

- **マージ:**
    - レビューでLGTM (Looks Good To Me) が出たら、**PRを作成した本人が** `main`ブランチにマージします。
    - マージが完了したら、作業ブランチは削除して構いません。

---

もしこのルールで分かりにくい点や、改善したい点があれば、いつでも話し合って更新していきましょう！
