# チーム開発の進め方とルール

このプロジェクトに貢献していただきありがとうございます！
円滑なチーム開発のために、以下のルールとガイドラインを守るようご協力をお願いします。

## 💬 基本的なコミュニケーション

- **相談:** 実装で迷ったり、技術的な問題に直面した場合は、一人で抱え込まずに気軽に相談しましょう。
- **情報共有:** すべてのタスクの進捗は、後述するGitHub Projectsのボード上で可視化されます。

---

## 🚀 開発ワークフロー (GitHub Issues & Projects)

このプロジェクトでは、すべての開発作業をGitHub IssuesとGitHub Projectsを使って管理します。以下のステップに沿って開発を進めてください。

### 1. タスクの起票 (Creating an Issue)

- **すべての作業はIssueから:** 新機能の追加、バグ修正、ドキュメントの更新など、いかなる作業も、まずは[リポジトリのIssuesタブ](./issues)でIssueを作成することから始めます。
- **Issueの作成後:** 作成されたIssueは、自動的に[GitHub Projects](./projects)の`Backlog`カラムに追加されます。

### 2. 開発の開始 (Starting Development)

1.  **Issueを担当する:** `Backlog`にあるIssueの中から、自分が担当するものを決め、自分自身を担当者（Assignee）に設定します。
2.  **ステータスを更新:** GitHub Projectsのボード上で、担当するIssueのカードを`Backlog`から`In Progress`に移動させます。これで、チームの誰もが「誰が何に取り組んでいるか」を把握できます。
3.  **ブランチを作成:** `main`ブランチから、Issue番号を含んだ名前で新しいブランチを作成します。

    ```bash
    # mainブランチを最新の状態にする
    git checkout main
    git pull origin main

    # 新しいブランチを作成して移動する (例: Issue番号が12の場合)
    git checkout -b feature/12-user-login
    ```

### 3. 実装とコミット (Implementation & Commits)

- 実装を進め、キリの良いタイミングでコミットを行ってください。
- コミットメッセージのルールについては、後述の「✅ コミットメッセージの書き方」セクションを参照してください。

### 4. Pull Requestとレビュー依頼 (PR & Review)

1.  **Pull Requestを作成:** 実装が完了したら、GitHub上で`main`ブランチへのPull Request (PR)を作成します。
2.  **Issueをリンクさせる:** PRの説明欄に`Closes #12`のように、関連するIssue番号をキーワード付きで記載します。これにより、PRがマージされると自動でIssueがクローズされるようになります。
3.  **ステータスを更新:** PRを作成したら、GitHub Projectsのボード上で、対応するIssueのカードを`In Progress`から`In Review`に移動させます。
4.  **レビューを依頼:** もう一人のメンバーに、コードレビューを依頼します。

### 5. マージと完了 (Merging & Completion)

1.  **レビューと修正:** レビューでの指摘事項を修正し、承認（Approve）をもらいます。
2.  **マージ:** 承認されたら、**PRを作成した本人が**`main`ブランチにマージします。
3.  **完了:** PRがマージされると、`Closes #12`のキーワードによってIssueが自動的に閉じられます。GitHub Projectsの設定により、閉じられたIssueは自動的に`Done`カラムに移動します。
4.  **ブランチを削除:** マージが完了した作業ブランチは削除して構いません。

## 🌿 ブランチ戦略

上記のワークフローで説明した通り、このプロジェクトでは「GitHub Flow」を採用しています。ブランチ作成のルールと命名規則の詳細は以下の通りです。

- **`main`ブランチは常に正常:** `main`ブランチは常にデプロイ可能な状態を保ちます。
- **ブランチ名:** `プレフィックス/Issue番号-目的`のように命名します。
    - **プレフィックス:** `feature/`, `fix/`, `docs/` などを付けます。
    - **例:**
        - `feature/12-user-login-page` （Issue #12 のユーザーログインページの開発）
        - `fix/23-header-layout-bug` （Issue #23 のヘッダーレイアウト崩れの修正）

## 🤖 コミット前の自動チェック (lint-staged & husky)

このプロジェクトでは、`git commit`を実行する際に、ステージングされたファイルに対して自動でコードのチェックとフォーマットが実行されます。

- **自動フォーマット (Prettier):** コードの見た目が自動で整形され、コミットに含められます。
- **品質チェック (ESLint):** 自動で修正できない問題（未使用の変数など）を検知した場合、**コミットは中止されます。** エラーを修正してから、再度コミットしてください。

---

## ✅ コミットメッセージの書き方

このルールは`commitlint`によって自動的にチェックされ、規約に違反したメッセージではコミットが失敗します。

- **`feat:`**: 新機能の追加
- **`fix:`**: バグの修正
- **`docs:`**: ドキュメントの変更
- **`style:`**: コードのフォーマット変更
- **`refactor:`**: リファクタリング
- **`test:`**: テストコードの追加・修正
- **`chore:`**: ビルドプロセスや補助ツールの変更
